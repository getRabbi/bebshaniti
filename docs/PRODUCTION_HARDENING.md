# Production hardening: images, imports, statements and reversals

This phase extends the existing Supabase Auth, tenant/RLS and append-only ledger model. It does not make completed sales mutable and it does not grant browser access to the service-role key.

## Product images

The existing private `product-media` bucket is used. Its limits are 5 MiB and JPEG, PNG or WebP. Browser uploads use the signed-in user's Supabase session and an object path shaped as:

```text
<organization_id>/products/<random_uuid>.<extension>
```

Storage RLS derives organization scope from the first path segment. Only catalog roles can insert, replace or delete objects. Authenticated organization members may read their tenant's objects. Product list responses contain a one-hour signed URL generated by the API; `products.image_path` remains the canonical value. The frontend never receives `SUPABASE_SERVICE_ROLE_KEY`.

Replacing an image writes the new object first, updates the tenant-constrained product record, and then removes the old object. Failed product creation or replacement attempts remove the newly uploaded object.

## Product import

The import accepts UTF-8 CSV and XLSX files up to 5 MiB and 5,000 data rows. `POST /api/v1/products/import/preview` parses and validates without writing business data. `POST /api/v1/products/import/commit` repeats validation and then imports with a savepoint per row.

Supported headers are `product_name / পণ্যের নাম`, `category / ক্যাটাগরি`, `unit / ইউনিট`, `buying_price / ক্রয় মূল্য`, `selling_price / বিক্রয় মূল্য`, `sku`, `barcode`, `brand / ব্র্যান্ড`, `opening_stock / ওপেনিং স্টক`, `low_stock_alert / কম স্টক সতর্কতা`, `wholesale_price / পাইকারি মূল্য`, `mrp`, `supplier / সাপ্লায়ার`, `rack / তাক`, and `expiry_date / মেয়াদ`.

Required fields are name, category, unit, buying price and selling price. Duplicate matching order is barcode, SKU, then normalized name. Modes are:

- `skip`: leave duplicates unchanged.
- `update`: update the matched product/variant and set imported opening stock through an adjustment movement.
- `create`: reject duplicate rows and create only unique rows.

Every committed batch creates an append-only `product_imports` record with totals and a bounded error summary. Opening stock always goes through `stock_movements`; inventory balances are never edited directly.

## Customer statements

`GET /api/v1/customers/{id}/statement` returns business/customer identity, all-time totals, date-filtered ledger rows with running balances, linked sales and collection history. When `date_from` is supplied, entries before it are represented as the opening balance. The admin route supports A4 and 80 mm browser print; dashboard navigation is excluded by print CSS.

## Sale returns and voids

Completed sale rows and sale items remain unchanged. A return inserts `sale_returns` and `sale_return_items`; a void inserts `sale_voids`. All event tables are append-only and audited.

- Returned tracked items create positive `sale_return` stock movements.
- A due portion creates a customer-ledger credit.
- A paid portion creates a refund payment and cashbook outflow.
- A void performs the same compensating entries for the full original sale.
- Cumulative returned quantity cannot exceed sold quantity.
- A sale with returns cannot be voided, and a voided sale cannot be returned.
- Reasons are mandatory and operations require backend permissions.

Reports exclude voids and net return values/quantities/profit from the original completed sale. Refund payments reduce payment and collection summaries.

## Deployment order

1. Take or verify the latest Supabase backup.
2. Run `npx supabase db push --dry-run`, review migrations `202607050008_role_values.sql`, `202607050009_production_hardening.sql`, and `202607050010_import_policy_hardening.sql`, then run `npx supabase db push`.
3. Run `npx supabase db lint --linked --level warning`.
4. Deploy `apps/api` so the new tables exist before reversal/import endpoints receive traffic.
5. Deploy `apps/web_admin`.
6. Verify health, login, organization context, sidebar reads, product search, a private image upload/remove, import preview, statement print and permission-denied behavior.

Environment variable names are unchanged. The API requires `DATABASE_URL`, `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_JWKS_URL` (or legacy JWT secret), audience and allowed origins. The admin requires only `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` and `NEXT_PUBLIC_API_BASE_URL`.
